# API Rules — NestJS Backend

## Stack

- Runtime: Node.js 20+
- Framework: NestJS with TypeScript strict mode
- Authentication: better-auth integrated directly into NestJS via `toNodeHandler`
- ORM: Drizzle ORM (`drizzle-orm` + `drizzle-kit`)
- Database: PostgreSQL 15 with PostGIS 3 extension
- Scheduler: `@nestjs/schedule` for cron jobs
- Validation: `class-validator` + `class-transformer` with global `ValidationPipe`
- Push notifications: Expo Push Notifications API
- File storage: Cloudflare R2 (S3-compatible)
- Deploy: Railway (Docker-based)
- HTTP client (internal): `axios` via `@nestjs/axios` if needed

## Authentication (Better-Auth)

Better-auth is mounted inside the NestJS application using `toNodeHandler`. All auth routes are exposed under `/api/auth/*`. The NestJS `AuthModule` wraps the better-auth instance and exposes it as a provider so other modules can inject it.

The `BetterAuthGuard` validates every request by calling `auth.api.getSession()`. It attaches the session user to the request object as `req.user`. Use the `@CurrentUser()` decorator to access the authenticated user inside controllers. Public routes must be decorated with `@Public()` which skips the guard.

Never implement custom JWT logic. Better-auth owns session management entirely.

```typescript
// Example controller usage
@Get('me')
getProfile(@CurrentUser() user: SessionUser) {
  return this.vehiclesService.findAllByUser(user.id);
}
```

## Drizzle ORM Rules

- All schema definitions live in `src/db/schema/`. One file per table (e.g. `vehicles.schema.ts`, `trips.schema.ts`).
- Always export the table definition and the inferred TypeScript type from each schema file.
- Use `drizzle-kit push` for development. Use `drizzle-kit generate` + `drizzle-kit migrate` for production migrations.
- Never write raw SQL strings for standard CRUD. Use Drizzle's query builder.
- For PostGIS operations (distance calculation, geometry queries), use `db.execute(sql`...`)` with Drizzle's tagged `sql` template. Always add a comment explaining why raw SQL is needed.
- Always use transactions (`db.transaction()`) when writing to multiple tables in a single operation.
- Never fetch more columns than needed. Always select explicitly rather than using `select()` with no arguments when returning data to the client.

```typescript
// Good — explicit column selection
const vehicles = await db
  .select({ id: vehicles.id, brand: vehicles.brand, model: vehicles.model })
  .from(vehiclesTable)
  .where(eq(vehiclesTable.userId, userId));

// Bad — fetches all columns including sensitive ones
const vehicles = await db.select().from(vehiclesTable);
```

## DTOs and Validation

Every incoming request body must have a corresponding DTO decorated with `class-validator` decorators. The global `ValidationPipe` is configured with `whitelist: true` and `forbidNonWhitelisted: true`, which means undeclared properties are automatically stripped and unknown properties cause a 400 error.

```typescript
export class CreateVehicleDto {
  @IsString()
  @MinLength(1)
  @MaxLength(100)
  brand: string;

  @IsString()
  @MinLength(1)
  @MaxLength(100)
  model: string;

  @IsInt()
  @Min(1900)
  @Max(new Date().getFullYear() + 1)
  year: number;

  @IsNumber()
  @Min(0)
  initialKm: number;
}
```

## Error Handling

Use NestJS built-in HTTP exceptions (`NotFoundException`, `BadRequestException`, `ForbiddenException`, etc.). Never throw raw `Error` objects from services — always use typed NestJS exceptions. The global exception filter formats all errors consistently.

```typescript
// Good
if (!vehicle) throw new NotFoundException('Vehicle not found');

// Bad
if (!vehicle) throw new Error('not found');
```

## Authorization

Always verify resource ownership before returning or mutating data. A user must never be able to access another user's vehicles, trips or maintenance records. This check happens in the service layer, not the controller.

```typescript
async findOne(id: string, userId: string): Promise<Vehicle> {
  const vehicle = await this.db.query.vehicles.findFirst({
    where: and(eq(vehicles.id, id), eq(vehicles.userId, userId)),
  });
  if (!vehicle) throw new NotFoundException('Vehicle not found');
  return vehicle;
}
```

## Trip Points (GPS Batch Endpoint)

The `POST /trips/:id/points` endpoint accepts an array of GPS points in a single request. This is intentional — the mobile client accumulates points locally and sends them in batches every 10 seconds to avoid overwhelming the API. The endpoint must validate the array size (max 100 points per batch) and insert all points in a single database transaction. Each point must contain `latitude`, `longitude`, `speedMps`, `altitudeM` and `recordedAt`.

After insertion, the endpoint must not recalculate trip stats synchronously. Stats (distance, avg speed, max speed, elevation gain) are calculated when the trip is finalized via `PATCH /trips/:id/finish`.

---

## Trip Finalization and PostGIS Stats

When `PATCH /trips/:id/finish` is called, the service must calculate all stats using PostGIS queries. Use `ST_MakeLine` to build the route geometry from ordered trip points. Use `ST_Length(geography)` to calculate total distance in meters. Calculate average and max speed from the `speedMps` column. Calculate elevation gain by summing positive altitude differences between consecutive points ordered by `recordedAt`.

Always store the calculated geometry in the `trips.routeGeography` column as a `GEOGRAPHY(LineString, 4326)` for future use (export, spatial queries).

## Scheduler — Maintenance Alerts

The `MaintenanceScheduler` runs a cron job once per day at 08:00 (America/Sao_Paulo timezone). It queries all maintenance records where `nextAlertKm` is within 200 km of the vehicle's `currentKm`, or where `nextAlertDate` is within 7 days of today. For each match, it sends a push notification via the `PushNotificationsService`. It must not send duplicate notifications — add an `alertSentAt` timestamp column to track the last notification sent per record.

## File Uploads (Cloudflare R2)

Vehicle photo uploads use `multer` with in-memory storage. The `UploadService` receives the buffer and uploads it to Cloudflare R2 using the S3-compatible SDK (`@aws-sdk/client-s3`). File size limit is 5MB. Accepted MIME types are `image/jpeg`, `image/png` and `image/webp`. The returned public URL is stored in `vehicles.imageUrl`.

## Response Shape

All successful responses must follow this shape:

```typescript
// Single resource
{ data: T }

// List
{ data: T[], total: number, page: number, limit: number }

// Mutation (create/update/delete)
{ data: T, message: string }
```

Use a global response interceptor to wrap responses automatically.

---

## Testing

Unit tests use Jest and `@nestjs/testing`. Every service must have a corresponding `.spec.ts` file covering the happy path and the main error cases. Tests must not connect to a real database — mock the Drizzle instance using `jest.fn()`. E2E tests use Supertest with a dedicated test database spun up via Docker Compose.

Minimum coverage requirements: services 70%, controllers 50%.

---

## Docker and Deploy

The `apps/api/Dockerfile` must use a multi-stage build. Stage 1 (`builder`) installs all dependencies and compiles TypeScript. Stage 2 (`runner`) copies only the compiled output and production dependencies. The final image must not contain `devDependencies`, source `.ts` files or the `node_modules/.cache` directory. The Railway service must expose port 3000 and define a health check at `GET /health`.
